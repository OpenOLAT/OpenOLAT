OpenOlat Architecture Reference

A comprehensive guide to the architecture, design patterns, and internal workings of the OpenOlat Learning Management System.

1. Overview & Project Structure

OpenOlat is a full-featured Learning Management System built as a monolithic Java web application. It follows a server-centric architecture — an established approach also used by frameworks like Vaadin, JSF, and Wicket — where the server maintains the complete UI state for each user session, renders HTML via Apache Velocity templates, and delivers incremental DOM updates over AJAX. The browser acts as a thin rendering layer, while all UI logic, state management, and security enforcement happen on the server.

This design has significant implications. Every user interaction — a button click, a dropdown change, a table sort — triggers a round-trip to the server, where the framework locates the target UI component, dispatches the event to the responsible controller, and re-renders only the parts of the page that changed. The server holds the entire component tree in memory, synchronized per browser window, ensuring that the UI state is always consistent and authoritative.

Technology Stack

  • Build: Maven (mvn compile -pl :openolat-lms -q)
  • Dependency Injection: Spring Framework 7 (XML + annotation-based)
  • ORM: Hibernate 7 / JPA via Jakarta Persistence
  • Templating: Apache Velocity (server-side rendering)
  • Frontend: Server-rendered HTML + custom AJAX partial update protocol
  • CSS: SASS-based theming with Bootstrap integration
  • Databases: MySQL, PostgreSQL, Oracle

Understanding these layers helps orient developers within the codebase:

  • Foundation provides the runtime infrastructure: the servlet container receives HTTP requests, the DispatcherModule routes them to the correct handler, Spring manages dependency injection across all modules, and the security layer enforces authentication and authorization. Configuration flows from olat.properties through Spring property placeholders into module beans.
  • Persistence abstracts all data storage. The VFS (Virtual File System) provides a uniform interface for file storage, whether files live on the local filesystem or are backed by metadata in the database.
  • Services are stateless Spring beans that encapsulate business operations. They are the primary integration point — controllers call services, services call the persistence layer. Services are accessed via CoreSpringFactory.getImpl(ServiceInterface.class) or via @Autowired injection.
  • Modules implement the LMS domain logic: courses with their node types and conditions, the assessment pipeline, the repository for learning resources, curriculum management, groups, portfolio, question pools, lectures with attendance tracking, and more. Each module is largely self-contained with its own Spring context, entities, services, and UI controllers. Modules live primarily in org.olat.modules.* and org.olat.course.*.
  • UI Layer is the custom MVC framework. Controllers manage user interaction, components model the UI widget tree, Velocity templates render HTML, and the AJAX engine delivers incremental updates. This layer is unique to OpenOlat and is documented in detail in the following sections.
  • API Layer exposes OpenOlat's functionality to external systems. The REST API (built on Apache CXF / JAX-RS) provides CRUD operations for most domain objects. WebDAV allows file-based access. LTI 1.3 enables tool integration with other LMS platforms. SCORM and IMS QTI 2.1 support standardized e-learning content and assessments.
Key principle: Dependencies flow downward. UI controllers depend on services, services depend on persistence, but never the reverse. This layering ensures that business logic is independent of the UI and can be accessed equally from REST endpoints, background jobs, or the web interface.

3. HTTP Request Lifecycle

Every user interaction in OpenOlat — whether clicking a button, submitting a form, or navigating to a new page — follows the same fundamental request lifecycle. Understanding this flow is essential for working with the framework, as it explains how a browser click travels through the system to update the UI.

The Big Picture

The request lifecycle has three main phases: routing (finding the right handler), dispatch (executing the business logic), and rendering (producing the response). The entire cycle is designed around server-side state: the server maintains the complete component tree for each browser window, and each request targets a specific component within that tree by its unique ID.

The dispatch phase is where the real work happens. The target component translates the HTTP request into a semantic event and fires it to its listening controller. The controller then executes business logic — calling services, updating data, loading models. Crucially, the controller may also fire events to parent controllers (e.g., fireEvent(ureq, Event.DONE_EVENT)), causing a chain of reactions up the controller hierarchy. It may also fire multi-user events via the EventBus to notify other user sessions. Only when this entire event chain has completed — all business logic executed, all models loaded, all component state updated — does the render phase begin. During rendering, the framework simply reads the already-prepared component state and produces HTML. No business logic, no service calls, and no database access should occur during rendering.

The windowID identifies which browser window/tab the request belongs to (OpenOlat supports multiple windows per session). The componentID identifies which UI widget was interacted with. The timestamps are version counters that prevent processing stale requests — if a user hits the back button and re-clicks a link, the timestamps will not match the current component tree, and the framework will reject the dispatch and re-render instead.

Phase 2: Dispatch (Event Chain)

Inside Window.dispatchRequest(), the framework searches the component tree for the target component by its dispatch ID, validates the timestamp (staleness check), verifies the CSRF token, and calls target.dispatchRequest(ureq).

The component translates the raw request into a semantic event and fires it to its listening controller. What follows is an event chain that may involve multiple controllers:

  1. Component → Controller: The target component fires an event. A Link fires Event("clicked"), a form fires FormEvent. The controller receives this in event(ureq, Component, Event).
  2. Business logic: The controller calls services, loads data, updates the database — all within its event() method.
  3. Controller → Parent Controller: If the work is complete, the controller may call fireEvent(ureq, Event.DONE_EVENT) to notify its parent. The parent receives this in event(ureq, Controller, Event) and may react — closing a dialog, refreshing a table, navigating to a different view.
  4. Multi-User Events: The controller may also fire events to the EventBus (fireEventToListenersOf(multiUserEvent, channel)) to notify other user sessions of the change.
  5. Component tree updates: Along the way, controllers modify component state (push data into VelocityContainers, add/remove children, change labels). Each modification sets the dirty flag on the affected component.

Only when the entire event chain has completed — all event() methods have returned — does the render phase begin.

Synchronization: The entire dispatch-render cycle runs synchronized(window). Concurrent AJAX requests to the same browser window are serialized, preventing race conditions. Different browser windows (tabs) have different Window instances and can process concurrently.

Although the database connection is technically still open (it is closed in the servlet's finally block after the response is sent), no DB access, no service calls, and no business logic should occur during the render phase. 4. Controller Hierarchy

Controllers are the workhorses of the OpenOlat UI. They receive user events, execute business logic (often by delegating to services), and update the component tree to reflect the new state. The framework provides a layered controller hierarchy, where each level adds progressively more convenience and structure.

Figure 3: Controller class hierarchy — each level adds convenience and structure

Controller (Interface)

The root Controller interface defines the contract: every controller has an initial component (the root of its UI subtree), can register listeners for controller-to-controller events, and must be disposable. This interface is intentionally minimal — it defines the communication protocol but leaves the implementation to subclasses.

DefaultController — Event Dispatch Engine

The first concrete superclass provides the event dispatch machinery. It defines two abstract event() methods corresponding to the two event channels (component-to-controller and controller-to-controller), and the fireEvent() method that propagates events upward to parent controllers. DefaultController also handles the disposed state: if a controller is disposed while a request is in flight, it gracefully replaces its content with a "stale" message rather than crashing.

// Two event channels every controller can handle:
protected abstract void event(UserRequest ureq, Component source, Event event);
protected void event(UserRequest ureq, Controller source, Event event) { }

// Fire an event upward to any listening parent controllers:
protected void fireEvent(UserRequest ureq, Event event) {
    for (ControllerEventListener listener : listeners) {
        listener.dispatchEvent(ureq, this, event);
    }
}

BasicController — Lifecycle Management

This is the most commonly extended controller class. It introduces the crucial listenTo() / removeAsListenerAndDispose() pattern that manages child controller lifecycles. When you call listenTo(childCtrl), the parent registers as a listener for events from the child AND tracks the child for automatic disposal. When the parent is disposed, all tracked children are disposed too. This prevents memory leaks and ensures clean teardown.

// The fundamental lifecycle pattern:
private EditController editCtrl;

// When creating a child controller, always listenTo():
editCtrl = new EditController(ureq, getWindowControl());
listenTo(editCtrl);

// When replacing a child controller, always dispose the old one first:
removeAsListenerAndDispose(editCtrl);
editCtrl = new EditController(ureq, getWindowControl());
listenTo(editCtrl);

FormBasicController — Structured Form Handling

For controllers that manage data entry forms, FormBasicController provides a structured lifecycle. Instead of manually handling component events, you implement initForm() to build the form using the FormUIFactory, validateFormLogic() to check inputs, and formOK() to process the submitted data. The form framework handles validation orchestration, error display, dirty tracking, and cancel/reset operations automatically.

How Controllers Compose

Complex screens are built by composing controllers hierarchically. A main controller creates child controllers, registers them via listenTo(), and embeds their components in its Velocity template. When a child needs to communicate with its parent (e.g., "the user saved the form"), it calls fireEvent(ureq, Event.DONE_EVENT), and the parent receives this in its event(ureq, Controller, Event) method.

5. Component Model

If controllers are the brains of the UI, components are the body. The component tree is an in-memory representation of the entire page structure for a given browser window. Each component knows how to render itself to HTML, tracks whether it needs re-rendering (the dirty flag), and can dispatch user interactions back to its listening controller.

The component model follows a composite pattern: containers hold children, and the renderer traverses the tree recursively. The key insight is that every UI widget on the page has a corresponding server-side component object, identified by a unique dispatch ID. This ID appears in the generated HTML (as a DOM element ID) and in the AJAX request URLs, allowing the framework to route any user interaction to the exact component that should handle it.

Figure 5: Component class hierarchy — a composite pattern for building UI trees

The Dirty Flag Mechanism

The dirty flag is the core optimization that makes server-side rendering performant. Instead of re-rendering the entire page on every interaction, the framework only re-renders components whose state has changed. When a controller modifies a component (e.g., calls contextPut() on a VelocityContainer, adds/removes children from a Container, or changes a Link's label), the component automatically sets its dirty flag to true. During the render phase, the framework traverses the tree, collects all dirty components, and renders only those — producing a compact JSON response with targeted DOM updates.

ComponentRenderer — Singleton per Type

Each component type has a corresponding ComponentRenderer implementation, held as a static singleton. The renderer is stateless and thread-safe — it receives the component, a StringOutput buffer, a URLBuilder (for generating framework URLs), and a Translator (for i18n). The VelocityContainerRenderer merges the template with the context, while simpler renderers like LinkRenderer emit HTML directly.

public interface ComponentRenderer {
    void render(Renderer renderer, StringOutput sb, Component source,
        URLBuilder ubu, Translator translator, RenderResult renderResult, String[] args);
}

// Each component type returns its singleton renderer:
private static final ComponentRenderer RENDERER = new VelocityContainerRenderer();
public ComponentRenderer getHTMLRendererSingleton() { return RENDERER; }

Window & WindowControl

The Window component is the root of each browser window's component tree. It owns the dispatch-render cycle: Window.dispatchRequest(ureq) finds the target component, executes the event chain, collects dirty components, renders them, and sends the response. The entire cycle runs synchronized(window), preventing concurrent dispatch on double-click or rapid navigation.

Controllers do not interact with the Window directly. Instead, they use WindowControl — a facade that provides:

  • Modal dialogs: pushAsModalDialog(component), pushAsTopModalDialog(component), pushAsCallout(component, targetId)
  • User messages: setInfo(msg), setWarning(msg), setError(msg) — displayed as growl-style notifications
  • Navigation: pop() to remove the current layer, makeFlat() to collapse the stack, pushFullScreen(controller) for full-screen views
  • Business path: getBusinessControl() for deep linking (see section 12)
  • Back office: getWindowBackOffice() to access the command queue, send JS commands to the browser, and register cycle listeners

The WindowBackOffice maintains a command queue that accumulates during the request. Controllers can send arbitrary commands to the browser — JS/CSS inclusions, redirects, scroll commands, business path updates — via wbo.sendCommandTo(command). The client-side JavaScript in functions.js processes this queue sequentially after receiving the AJAX response.

6. Event System

OpenOlat has three distinct event mechanisms, each serving a different scope and purpose. Understanding when to use each is fundamental to working with the framework correctly.

Figure 6: Three event mechanisms — component events (UI), controller events (parent-child), multi-user events (cross-session)

6.1 Component Events

When a user clicks a link or interacts with a UI widget, the component translates the raw HTTP request into a semantic event and fires it to all registered ComponentEventListeners (typically one controller). The event is delivered synchronously within the same request cycle.

// Inside AbstractComponent — fires to all listening controllers:
protected void fireEvent(final UserRequest ureq, final Event event) {
    for (ComponentEventListener listener : listeners) {
        listener.dispatchEvent(ureq, this, event);  // → controller.event(ureq, Component, Event)
    }
}

// VelocityContainer dispatches command string as event:
protected void doDispatchRequest(UserRequest ureq) {
    String cmd = ureq.getParameter(COMMAND_ID);
    fireEvent(ureq, new Event(cmd));
}

The controller identifies the source by reference comparison:

protected void event(UserRequest ureq, Component source, Event event) {
    if (source == editLink) { /* handle edit click */ }
    if (source == deleteLink) { /* handle delete click */ }
}

6.2 Controller Events

Controller events flow upward from child to parent. When a child controller finishes its task, it calls fireEvent(ureq, Event.DONE_EVENT), and the parent receives this in its event(ureq, Controller, Event) method. This is the primary mechanism for decoupling controllers — the child doesn't know who its parent is; it just fires a semantic event.

// Child fires upward (in formOK, for example):
fireEvent(ureq, Event.DONE_EVENT);

// Parent receives (wired via listenTo()):
protected void event(UserRequest ureq, Controller source, Event event) {
    if (source == editCtrl && event == Event.DONE_EVENT) {
        // Close modal, refresh table, etc.
        getWindowControl().removeModalDialog(editCtrl.getInitialComponent());
        loadTableData();
    }
}

Standard event constants: DONE_EVENT, CANCELLED_EVENT, CHANGED_EVENT, BACK_EVENT, CLOSE_EVENT, FAILED_EVENT.

6.3 Form Events

The FlexiForm system extends the component event mechanism with FormEvent, which carries both the source FormItem and a trigger type indicating how the user interacted with the element. Form elements that call addActionListener(FormEvent.ONCHANGE) will fire events immediately on interaction, rather than waiting for form submission.

public class FormEvent extends Event {
    public static final int ONCLICK = 2;
    public static final int ONCHANGE = 4;
    public static final int ONBLUR = 16;
    // ...
    private final FormItem source;
    private final int trigger;
}

// FormBasicController routes form events internally:
if (event == Form.EVNT_VALIDATION_OK) {
    formOK(ureq);                              // form submitted, valid
} else if (event instanceof FormEvent fe) {
    formInnerEvent(ureq, fe.getFormItemSource(), fe);  // element event
}

In formInnerEvent(), you can react to individual element changes:

protected void formInnerEvent(UserRequest ureq, FormItem source, FormEvent event) {
    if (source == statusDropdown) {
        // Dropdown changed — update dependent fields
        updateVisibility(statusDropdown.getSelectedKey());
    } else if (source == previewLink) {
        // Button clicked
        doPreview(ureq);
    }
}

6.4 Multi-User Events (EventBus)

The first two event channels are single-user, single-request mechanisms. But OpenOlat also needs to communicate across user sessions — for example, when an author publishes a course change, all users currently viewing that course need to be notified. This is the role of the EventBus.

The EventBus uses a publish/subscribe pattern with OLATResourceable as the channel identifier. An OLATResourceable is simply a pair of (typeName, id) that uniquely identifies a resource — a course, a repository entry, a chat room, etc. Any component (controller, service, or background job) can register as a listener on a channel and will receive MultiUserEvents fired to that channel by any user session, even across cluster nodes.

// OLATResourceable identifies the event channel:
public interface OLATResourceable {
    String getResourceableTypeName();  // e.g., "CourseModule"
    Long getResourceableId();          // e.g., 123456L
}

// Create a channel identifier:
OLATResourceable courseOres = OresHelper.createOLATResourceableInstance("CourseModule", courseId);

// Register to receive events (typically in controller constructor):
coordinatorManager.getCoordinator().getEventBus()
    .registerFor(this, getIdentity(), courseOres);

// Fire an event to all listeners on the channel:
coordinatorManager.getCoordinator().getEventBus()
    .fireEventToListenersOf(new MultiUserEvent("courseChanged"), courseOres);

// Deregister when done (typically in dispose):
coordinatorManager.getCoordinator().getEventBus()
    .deregisterFor(this, courseOres);

The listening controller implements GenericEventListener:

public interface GenericEventListener {
    void event(Event event);  // no UserRequest — async, cross-session
}

// In the controller:
@Override
public void event(Event event) {
    if (event instanceof MultiUserEvent mue) {
        if ("courseChanged".equals(mue.getCommand())) {
            // Mark component dirty so it re-renders on next poll
            mainVC.setDirty(true);
        }
    }
}
Key differences: Component and controller events are synchronous, within the same HTTP request, and carry a UserRequest. Multi-user events are asynchronous (delivered on the next AJAX poll), carry no UserRequest, and are serializable for cluster communication. MultiUserEvents should be fire-and-forget — no veto, no response.

There is also a per-session EventBus (ureq.getUserSession().getSingleUserEventCenter()) for intra-session communication between controllers that don't have a parent-child relationship — for example, toggling a glossary overlay from a toolbar button.

Real-World Example: CourseRuntimeController

When a user enters a course, the CourseRuntimeController registers on the course's EventBus channel to receive notifications about course changes, assessment mode changes, and more:

// On entry — register for course and repository events:
coordinatorManager.getCoordinator().getEventBus()
    .registerFor(this, getIdentity(), getOlatResourceable());
coordinatorManager.getCoordinator().getEventBus()
    .registerFor(this, getIdentity(), getRepositoryEntry());

// Notify others that we joined:
coordinatorManager.getCoordinator().getEventBus()
    .fireEventToListenersOf(new MultiUserEvent("joined"), getOlatResourceable());

// On exit — clean up:
coordinatorManager.getCoordinator().getEventBus()
    .fireEventToListenersOf(new MultiUserEvent("left"), getOlatResourceable());
coordinatorManager.getCoordinator().getEventBus()
    .deregisterFor(this, getOlatResourceable());

7. Velocity Templating

OpenOlat uses Apache Velocity for server-side HTML rendering. Templates define the visual structure, while controllers provide the data and child components. The framework enhances Velocity with the $r decorator object — a VelocityRenderDecorator that bridges the template and the component framework.

Templates live in _content/ directories alongside the controller's Java package. When a controller calls createVelocityContainer("mypage"), the framework resolves the template path automatically from the controller's package: org/olat/modules/feature/ui/_content/mypage.html.

The $r Decorator

The $r object is the most important tool in templates. It provides:

## Render a child component (controller or form element)
$r.render("editForm")
$r.render("dataTable")

## Translate an i18n key
$r.translate("page.title")

## Get the DOM ID of a component (for JavaScript targeting)
$r.getId("myLink")

## Conditional rendering based on component visibility
#if($r.visible("optionalSection"))
    $r.render("optionalSection")
#end

## Access context data pushed by the controller
<h2>$userName</h2>
#foreach($item in $items)
    <div class="o_item">$item.name</div>
#end

The separation is clean: the controller pushes data via contextPut() and adds child components via put(); the template decides layout and presentation. Controllers never emit HTML, and templates never contain business logic.

Complete $r Method Reference

The $r decorator (VelocityRenderDecorator) exposes rendering, translation, formatting, and link-generation utilities to templates. Formatter and StringHelper are not directly exposed as variables — their functionality is available through $r methods:

10. AJAX Update Cycle

OpenOlat's AJAX mechanism is what makes the server-centric architecture feel responsive. Rather than sending full page reloads, the framework sends only the HTML fragments that changed. This section ties together the concepts introduced earlier into a complete picture of how a user click results in a partial page update.

Figure 8: AJAX update cycle — only dirty components are re-rendered and sent as JSON

The WindowBackOfficeImpl maintains a command queue that accumulates during the request. Besides DOM replacement commands, the queue can contain JS/CSS inclusion commands, redirect commands, scroll commands, and more. The client-side JavaScript in functions.js processes this queue sequentially, applying each command to the page.

Performance implication: The dirty-flag mechanism means that even complex pages with hundreds of components can update efficiently. If a user clicks a "save" button that updates a status label and refreshes a counter, only those two small components are re-rendered and sent over the wire — not the entire page with its navigation, tables, and sidebars.

The Poller — Server-Push via Periodic Polling

User clicks trigger AJAX requests, but what about changes from other users? A chat message, a collaborative edit, a notification — these mark components as dirty on the server, but nobody clicks to fetch the update. The poller solves this with periodic background requests.

The poller is initialized per window by the AjaxController (via serverpart.html). It uses jQuery.periodic to schedule POST requests to a mapper URL:

  • Initial interval: 5 seconds (DEFAULT_POLLPERIOD in AjaxController)
  • Exponential decay: 1.005× per cycle — the interval grows slowly when nothing changes
  • Activity reset: Any mouseover, click, or keypress resets the interval to the base period
  • Inactivity timeout: Polling stops after 60 minutes of no user activity

On the server, the poll handler in AjaxController validates the CSRF token, then calls window.handleDirties() — the same method used after user-initiated AJAX requests. If dirty components exist, they are rendered and returned as JSON. If nothing changed, a 304 Not Modified response is returned (via NothingChangedMediaResource), and the client-side decay increases the next poll interval.

// AjaxController poll handler (simplified):
wboImpl.fireCycleEvent(Window.BEFORE_INLINE_RENDERING);
Command updateDirtyCom = window.handleDirties();
wboImpl.fireCycleEvent(Window.AFTER_INLINE_RENDERING);

if (updateDirtyCom != null) {
    // Dirty components found — return their rendered HTML as JSON
    resource = extractMediaResource(false);
} else {
    // Nothing changed — 304 triggers interval growth on client
    resource = new NothingChangedMediaResource();
}

The full cycle for server-push: another user's action fires a MultiUserEvent via the EventBus → the listening controller's event() method runs and calls setDirty(true) on the affected component → the next poll finds the dirty component via handleDirties() and sends the re-rendered HTML to the browser.

Registering Mappers in Controllers

BasicController provides convenience methods that automatically track mappers for cleanup:

// Non-cacheable — new random URL each time (browser always fetches fresh)
String url = registerMapper(ureq, (relPath, request) -> {
    return new FileMediaResource(myFile);
});

// Cacheable — stable URL derived from ID (enables browser caching)
String url = registerCacheableMapper(ureq, "gallery-" + galleryId,

// Cacheable with explicit expiration (seconds)
String url = registerCacheableMapper(ureq, "thumbnail-" + id, mapper, 3600);

// Template: <img src="$mapperUrl/image.jpg" />

Cacheable vs. Non-Cacheable Mappers

  • Non-cacheable (registerMapper): URL contains a random UUID hash. Each registration creates a new URL, forcing the browser to fetch fresh. Use for one-time or dynamic content.
  • Cacheable (registerCacheableMapper): URL is derived from a stable ID string via MD5. The same ID always produces the same URL, enabling browser caching with Last-Modified headers. Use for images, static resources, and content that changes infrequently.

Persisted Mappers (Cluster Support)

If a mapper implements Serializable, the MapperService automatically persists it to the database (via MapperDAO and the o_mapper table, serialized with XStream). This ensures that in a cluster, any node can serve the mapper's content. A scheduled MapperZombieSlayerJob periodically removes expired persisted mappers.

Global Mappers

For application-wide shared resources (JS translation maps, static generated content), use the GlobalMapperRegistry:

// Register a global mapper — URL includes build ID for cache-busting across releases

    .register(JSTranslatorMapper.class, jsTranslationMapper);
// Results in URL: /g/{buildId}org.olat.core.gui.JSTranslatorMapper/...

Mapper Cleanup

  1. Controller dispose — BasicController.doPreDispose() calls MapperService.cleanUp(mapperKeys) for all registered mappers
  2. Session destroy — MapperSessionListener cleans up all mappers when an HTTP session ends
  3. Zombie slayer — scheduled job removes persisted mappers older than 6 hours

12. Business Path & Deep Linking

A server-centric architecture faces a fundamental tension: the browser's URL bar is document-oriented (each URL identifies a page), but the server's UI state is component-oriented (a tree of controllers managing an interactive session). When a user clicks a link inside OpenOlat, the interaction is dispatched to a specific component via its internal ID — the resulting URL (/auth/1:2:3:4/...) is a component address, meaningful only to the current session. It cannot be bookmarked, shared, or opened in a new tab.

The business path system solves this problem. It provides a second, document-style address for every UI state — a human-readable, serializable path like [RepositoryEntry:123][CourseNode:456] that describes what the user is looking at, not which component is rendering it. This enables bookmarking, deep linking, browser back/forward navigation, "open in new window", and session resumption.

12.1 The Business Path Format

A business path is a sequence of ContextEntry segments, each identifying a resource by type and ID:

[RepositoryEntry:8519168][CourseNode:107803373834]
[RepositoryEntry:123][CourseNode:456][path=/documents/report.pdf:0]
[HomeSite:1234][notifications:0]

Each segment wraps an OLATResourceable — the same type/key pair used for EventBus channels (see section 6.4). The path reads left to right from the outermost resource (e.g., a repository entry) to the innermost (e.g., a file in a course folder).

12.2 How It Works: The Dual-Address Architecture

The system works bidirectionally:

Outbound (state → URL): After each user interaction, the active controller updates its BusinessControl with the current ContextEntry. During the AJAX render cycle, Window.handleBusinessPath() serializes the full business path to a URL and sends an AJAX command that updates the browser's URL bar via history.replaceState(). The user sees a clean, bookmarkable URL without a page reload.

Inbound (URL → state): When a user navigates to a business path URL (via bookmark, deep link, or "open in new tab"), the AuthenticatedDispatcher parses it into a list of ContextEntry objects and hands them to NewControllerFactory.launch(). The factory looks up a ContextEntryControllerCreator for the first resource type, creates the appropriate controller, and — if the controller implements Activateable2 — calls activate(ureq, remainingEntries, state) to restore the full navigation state recursively.

12.3 The Activateable2 Contract

Controllers that support deep linking implement the Activateable2 interface:

public interface Activateable2 {
    void activate(UserRequest ureq, List<ContextEntry> entries, StateEntry state);
}

The entries list contains the remaining context entries to process. A typical implementation peels off the first entry, navigates to the corresponding sub-view, and passes the rest to the child controller:

@Override
public void activate(UserRequest ureq, List<ContextEntry> entries, StateEntry state) {
    if (entries == null || entries.isEmpty()) return;

    ContextEntry entry = entries.get(0);
    String type = entry.getOLATResourceable().getResourceableTypeName();
    if ("CourseNode".equals(type)) {
        Long nodeId = entry.getOLATResourceable().getResourceableId();
        selectCourseNode(ureq, nodeId);  // Navigate to the node

        // Pass remaining entries to the child
        if (entries.size() > 1 && currentNodeCtrl instanceof Activateable2 a2) {
            a2.activate(ureq, entries.subList(1, entries.size()), null);
        }
    }
}

12.4 BusinessControl Chain

The BusinessControl interface models the business path as a linked list, mirroring the controller hierarchy. Each WindowControl in the controller tree wraps a StackedBusinessControl that points to its parent. When a controller updates its context entry, the full chain can be serialized top-down to produce the complete business path.

Key operations:

  • getAsString() — recursively serializes the full path from root: [Type:Id][Type:Id]...
  • popLauncherContextEntry() — during inbound activation, pops the next entry to process
  • setCurrentContextEntry(ce) — during outbound, updates the current segment

The factory method BusinessControlFactory.createBusinessWindowControl(contextEntry, parentWControl) creates a new layer in both the WindowControl and BusinessControl chains simultaneously.

12.5 NewControllerFactory & ContextEntryControllerCreator

NewControllerFactory is the central registry that maps resource types to controller creators. Each resource type (e.g., "RepositoryEntry", "HomeSite", "CatalogEntry") registers a ContextEntryControllerCreator that knows how to:

  • Create a controller for that resource type
  • Validate that the context entry is accessible to the current user
  • Decide whether to open the resource as a site (fixed tab) or a dynamic tab

The launch process integrates with the tab system (DTabs): resources open as tabs in the main UI, and activating an already-open tab simply calls activate() on its controller rather than creating a new one.

12.6 Session Resumption

The business path also powers session resumption. The HistoryManager persists the user's last navigation point (as a HistoryPoint containing the business path and context entries). On the next login, the ResumeController can offer to restore the previous state by re-launching the saved business path through the same NewControllerFactory.launch() mechanism.

Key Insight: The business path is what makes a server-centric application feel like a document-oriented web application. Without it, every URL would be a session-specific component address that breaks on refresh. With it, users get bookmarkable URLs, working back/forward buttons, and "open in new tab" — all while the server maintains full control of the UI state.

13. Controller Disposal & Housekeeping

In a server-centric architecture where UI state lives on the server, proper cleanup is critical to prevent memory leaks. Every controller, mapper, listener, and EventBus subscription consumes server memory. The framework provides a hierarchical disposal mechanism that automatically cleans up the controller tree.

The Dispose Lifecycle

// DefaultController.dispose() — called when controller is no longer needed
public synchronized void dispose() {
    synchronized (DISPOSE_LOCK) {
        if (disposed) return;    // prevent double-dispose
        disposed = true;          // suppress all future event dispatch
    }
    doPreDispose();   // framework cleanup (mappers, children)
    doDispose();      // custom cleanup (override this)
}

Once disposed = true, the controller silently ignores all incoming events. If a user somehow triggers a request to a disposed controller (e.g., stale browser tab), the framework displays a "page expired" message via the wrapperPanel.

Automatic Cleanup (BasicController.doPreDispose)

The framework handles these automatically — no developer action needed:

  • Child controllers registered via listenTo() — recursively disposed
  • Mappers registered via registerMapper() / registerCacheableMapper() — deregistered from MapperService
  • Disposable form items (e.g., FileElement with temp files) — FormBasicController disposes all Disposable form items

Manual Cleanup (Override doDispose)

Developers must handle cleanup for resources the framework doesn't track:

@Override
protected void doDispose() {
    // 1. Deregister EventBus listeners
    CoordinatorManager.getInstance().getCoordinator().getEventBus()
        .deregisterFor(this, courseOres);

    // 2. Release locks
    if (lockEntry != null && lockEntry.isSuccess()) {
        CoordinatorManager.getInstance().getCoordinator().getLocker()
            .releaseLock(lockEntry);
        lockEntry = null;
    }

    // 3. Null heavy references to help GC
    currentNodeController = null;
    largeDataStructure = null;
}

Replacing Controllers at Runtime

When a controller reference is replaced (e.g., switching views in a tab), always use removeAsListenerAndDispose():

// CORRECT: removes from child list, deregisters listener, disposes
removeAsListenerAndDispose(detailCtrl);
detailCtrl = new DetailController(ureq, wControl, item);
listenTo(detailCtrl);

// WRONG: leaks the old controller (still in child list, still listening)
detailCtrl = new DetailController(ureq, wControl, item);

15. Database Access (Hibernate/JPA)

OpenOlat's persistence layer wraps Hibernate/JPA behind a DB facade that provides thread-local EntityManager access. This design means any code — in a controller, service, or background job — can call DBFactory.getInstance().getCurrentEntityManager() to get the current transaction's entity manager without explicit dependency injection.

The framework manages transaction boundaries automatically during request processing: the request begins a transaction, controllers and services do their work, and the framework commits after dispatch. For background tasks, developers must explicitly call db.commitAndCloseSession().

// Typical service method pattern:
public List<MyEntity> findByName(String name) {
    return dbInstance.getCurrentEntityManager()
        .createQuery("select e from MyEntity e where e.name = :name", MyEntity.class)
        .setParameter("name", name)
        .getResultList();
}

public void save(MyEntity entity) {
    dbInstance.getCurrentEntityManager().persist(entity);
}

Entity Mapping

All entities use JPA annotations and are registered in src/main/resources/META-INF/persistence.xml (~400+ entity classes). The infrastructure includes HikariCP connection pooling, Infinispan L2 cache, and support for MySQL, PostgreSQL, and Oracle.

16. Hibernate Session & Entity Lifecycle

OpenOlat uses a strict session-per-request pattern. Each HTTP request gets its own Hibernate EntityManager via a ThreadLocal in DBImpl. The session is lazily created on first database access and closed at the end of the request. Between requests (between clicks), no Hibernate session exists — all entities from the previous request are detached.

The DB Lifecycle

// The DB instance is thread-local — always get it fresh, never cache it
@Autowired
private DB dbInstance;

// Typical service method:
public MyEntity createItem(String name) {
    MyEntity entity = new MyEntity();
    entity.setName(name);
    dbInstance.getCurrentEntityManager().persist(entity);
    dbInstance.commitAndCloseSession();   // commit + close in one call
    return entity;
}
MethodPurpose
getCurrentEntityManager()Get (or lazily create) the thread-bound EntityManager
commitAndCloseSession()Commit transaction, close EM, remove from ThreadLocal. Preferred method.
rollbackAndCloseSession()Rollback and close on error
intermediateCommit()Commit + close in bulk loops to avoid huge transactions

Preventing Stale Objects Between Requests

Since entities are detached after each request's session closes, the next request must reload them. Key patterns:

// Pattern 1: Simply reload by key (most common)
RepositoryEntry entry = repositoryEntryDAO.loadByKey(entryKey);

// Pattern 2: Detach stale entity, then reload fresh for update
dbInstance.getCurrentEntityManager().detach(staleEntry);
RepositoryEntry fresh = loadByKey(staleEntry.getKey());
dbInstance.getCurrentEntityManager().lock(fresh, LockModeType.PESSIMISTIC_WRITE);

// Pattern 3: Merge a detached entity back into current session
RepositoryEntry managed = dbInstance.getCurrentEntityManager().merge(detachedEntry);
dbInstance.commit();

Preventing LazyInitializationException

Since sessions close at request end, lazy-loaded associations must be fetched while the session is open:

// Use join fetch in JPQL queries:
String query = "select v from RepositoryEntry as v"
    + " inner join fetch v.olatResource as ores"    // eager
    + " left join fetch v.lifecycle as lifecycle"    // nullable, still eager
    + " where v.key = :key";

// Or force initialization by accessing the relationship:
RepositoryEntry entry = dao.updateAndCommit(re);
entry.getOlatResource().getResourceableTypeName();  // forces load
entry.getStatistics().getLaunchCounter();            // forces load

Avoiding Unintended Dirty Checking

Hibernate automatically detects changes to managed entities and flushes them on commit. To prevent accidental updates:

  • Don't modify managed entities unless you intend to persist the change
  • Use DTOs or value objects when passing data to the UI layer if modifications are possible
  • Detach entities (em.detach(entity)) if you need a read-only copy

Bulk Operations

For large batch operations, use intermediateCommit() to prevent huge transactions and excessive memory from accumulated dirty entities:

for (Item item : largeList) {
    process(item);
    if (counter++ % 100 == 0) {
        dbInstance.intermediateCommit();  // commits, closes session, next access creates new one
    }
}

DBImpl logs a warning if more than 500 database accesses occur in a single transaction, which typically indicates a missing intermediateCommit().

17. Virtual File System (VFS)

OpenOlat does not access the filesystem directly. Instead, all file operations go through the Virtual File System (VFS) — an abstraction layer that decouples application code from physical storage. The VFS provides a uniform API for reading, writing, copying, and deleting files, while transparently handling security checks, quota enforcement, metadata tracking, versioning, and trash management.

Files are physically stored on disk under the bcroot/ directory (the "base canonical root"), while metadata is tracked in the o_vfs_metadata database table. This separation means even read-only or archived files can carry rich metadata (author, title, download count, etc.) without modifying the filesystem.

17.1 Core Interfaces

The VFS is built on three interfaces forming a simple hierarchy:

  • VFSItem — base interface for both files and folders. Provides resolve(path) for navigation, rename(), delete(), exists(), capability checks (canRename(), canDelete(), canMeta(), canVersion()), and access to getMetaInfo() and getLocalSecurityCallback().
  • VFSContainer (extends VFSItem) — represents directories. Adds getItems() / getItems(filter) to list children, getDescendants(filter) for recursive listing, createChildContainer(name) / createChildLeaf(name) for creating children, and copyFrom(source, identity) for copying files into the folder (with quota checking).
  • VFSLeaf (extends VFSItem) — represents files. Adds getInputStream(), getOutputStream(append), and getSize().

17.2 Key Implementations

LocalFolderImpl and LocalFileImpl wrap java.io.File objects pointing into the bcroot/ directory. They are the workhorses — most VFS operations ultimately hit these classes. Both extend LocalImpl, which handles security callback inheritance and capability checks.

NamedContainerImpl is a transparent wrapper that gives a container a different display name without changing the underlying filesystem path. This is essential for virtual folder structures where the same physical folder appears under a user-friendly label.

MergeSource is one of the most powerful VFS concepts. It creates a virtual container that merges children from multiple unrelated physical folders into a single unified view. It has two modes of adding children:

// Mode 1: Add as a named sub-folder
MergeSource root = new MergeSource(null, "courseFiles");
root.addContainer(sharedFolder);        // appears as child folder

// Mode 2: Merge children directly into the root
root.addContainersChildren(courseFolder, true);  // enableWrite=true → write target
// courseFolder's children appear directly in root, not nested

The enableWrite=true parameter designates a folder as the rootWriteContainer — when a user uploads a file to this MergeSource, it lands in that folder. A MergeSource can have at most one write target.

VirtualContainer is a simple in-memory container (not backed by filesystem, not writable) for assembling ad-hoc collections of VFS items.

17.3 Security Callbacks

Access control is decentralized: each VFS item can carry a VFSSecurityCallback that defines what operations are permitted. The callback interface includes:

public interface VFSSecurityCallback {
    boolean canRead();
    boolean canWrite();
    boolean canCreateFolder();
    boolean canDelete();
    boolean canList();
    boolean canCopy();
    boolean canDeleteRevisionsPermanently();
    Quota getQuota();
}

Security callbacks inherit down the tree: a child item without its own callback inherits its parent's. This means setting a callback on a top-level container applies to the entire subtree. The method VFSManager.findInheritingSecurityCallbackContainer(item) walks up the tree to find the effective callback.

Common implementations:

  • FullAccessCallback — all operations permitted, no quota. Used for unrestricted system folders.
  • ReadOnlyCallback — read and list only. Used for decommissioned courses, published resources, shared content.
  • FullAccessWithQuotaCallback — full access with storage limits.
  • DefaultVFSSecurityCallback — everything denied. Safe fallback.

17.4 Metadata & Versioning

File metadata is managed by VFSRepositoryService and stored in the o_vfs_metadata database table. Metadata is separate from the file itself — the VFSMetadata interface exposes:

  • Identity tracking: getFileInitializedBy(), getFileLastModifiedBy(), getDeletedBy()
  • Dublin Core fields: title, creator, publisher, language, source, URL
  • Statistics: getDownloadCount(), getFileSize()
  • Deletion tracking: isDeleted(), getDeletedDate() for trash management

Versioning is automatic for files where canVersion() returns YES. When a file is saved via VFSRepositoryService.itemSaved(leaf, identity), a new revision is created. Revisions are stored as separate files with metadata in the database.

Trash: calling delete() on a VFS item does not permanently remove it. Instead, the file is moved to a ._ootrash folder within the same parent directory, and its metadata is marked as deleted. deleteSilently() bypasses this and removes permanently. Trash cleanup is handled by VFSRepositoryService.cleanTrash().

17.5 Filtering

The VFSItemFilter interface (single method accept(VFSItem)) controls which items appear when listing a container. Filters can be passed to getItems(filter) or set as default on a container via setDefaultItemFilter().

Built-in filters include VFSSystemItemFilter (excludes hidden files and __MACOSX), VFSLeafFilter (files only), VFSContainerFilter (folders only), VFSItemSuffixFilter (by extension), and composites like VFSAndFilter / VFSOrFilter.

17.6 Quota Management

Storage quotas are enforced through security callbacks. A Quota object specifies getQuotaKB() (maximum storage) and getUlLimitKB() (per-upload limit). The constant Quota.UNLIMITED = -1 disables enforcement.

Quota is checked automatically during copyFrom() operations in LocalFolderImpl. You can also check programmatically:

long quotaLeftKB = VFSManager.getQuotaLeftKB(container);
if (quotaLeftKB != Quota.UNLIMITED && quotaLeftKB < (fileSize / 1024)) {
    // Not enough space
}

17.7 VFSManager Utilities

The VFSManager class provides essential static helpers:

  • Path resolution: resolveFile(rootContainer, path), resolveOrCreateContainerFromPath(base, relPath), resolveOrCreateLeafFromPath(base, relPath)
  • Root access: olatRootContainer(relPath) and olatRootLeaf(relPath) create VFS items from paths relative to bcroot/
  • Security helpers: findInheritingSecurityCallback(item) walks up the tree; isInPath(path) prevents directory traversal attacks
  • Real path: getRealPath(container) unwraps NamedContainerImpl and MergeSource to get the actual filesystem path

17.8 HTTP File Delivery

VFSMediaResource bridges VFS and HTTP: it wraps a VFSLeaf and implements the MediaResource interface for browser delivery. It handles MIME types, Content-Disposition headers, range requests for streaming, and cache control. For security, HTML and SVG files receive restrictive CSP headers.

// Deliver a file for download
VFSLeaf file = (VFSLeaf) container.resolve("report.pdf");
VFSMediaResource resource = new VFSMediaResource(file);
resource.setDownloadable(true);  // Force download instead of inline
ureq.getDispatchResult().setResultingMediaResource(resource);

17.9 Real-World Example: MergedCourseContainer

The MergedCourseContainer demonstrates the full power of the VFS. It creates a unified view of a course's file resources by merging multiple sources:

public class MergedCourseContainer extends MergeSource {
    private void init(PersistingCourseImpl course) {
        if (courseReadOnly) {
            setLocalSecurityCallback(new ReadOnlyCallback());
        }

        // Course folder is the write target
        VFSContainer courseFolder = course.getIsolatedCourseFolder();
        addContainersChildren(courseFolder, true);  // merge + writable

        // Shared folder appears as a named sub-folder
        VFSContainer sharedFolder = getSharedFolder(sharedResource);
        addContainer(new NamedContainerImpl("_sharedfolder", sharedFolder));

        // Course node folders (file uploads, etc.)
        MergedCourseElementDataContainer nodes =
            new MergedCourseElementDataContainer(courseEntry, identityEnv);
        if (!nodes.isEmpty()) {
            addContainer(nodes);
        }
    }
}

From the user's perspective, all these sources appear as a single folder tree — the VFS handles the routing transparently.

Key Insight: The VFS is not just a file abstraction — it's a security and lifecycle layer. Never bypass it by accessing bcroot/ directly. Always go through VFSManager, VFSContainer, and VFSLeaf to ensure security callbacks, quota limits, metadata tracking, and versioning are respected.

18. Spring & Dependency Injection

OpenOlat uses Spring for dependency injection, with a hybrid approach: newer code uses annotation-based wiring (@Service, @Autowired), while older code uses XML configuration. The main Spring context at src/main/java/org/olat/_spring/mainContext.xml imports 70+ module-specific contexts, each located in their module's _spring/ directory.

In non-Spring-managed code (e.g., controllers, which are created by other controllers rather than by Spring), services are accessed via the service locator pattern:

// Service locator (in controllers and non-Spring classes):
UserManager userManager = CoreSpringFactory.getImpl(UserManager.class);

// Standard injection (in Spring-managed beans):
@Service
public class MyServiceImpl implements MyService {
    @Autowired private DB dbInstance;
    @Autowired private UserManager userManager;

Configuration properties flow from src/main/resources/serviceconfig/olat.properties (defaults) through olat.local.properties (overrides, not committed) into Spring beans via @Value annotations or property placeholders.

19. Scheduler & Background Tasks

OpenOlat uses Quartz for all periodic/scheduled tasks — no Spring @Scheduled annotations are used. The scheduler is configured in schedulerContext.xml with a 5-thread pool.

JobWithDB

All scheduled jobs extend JobWithDB, which automatically manages the database lifecycle:

public abstract class JobWithDB extends QuartzJobBean {
    protected final void executeInternal(JobExecutionContext context) {
        try {
            executeWithDB(context);              // subclass implements this
            DBFactory.getInstance().commitAndCloseSession();
        } catch (Exception e) {
            DBFactory.getInstance().rollbackAndCloseSession();
        }
    }
}

TaskExecutorManager

For asynchronous (non-periodic) task execution, TaskExecutorManager provides multiple thread pools:

  • mpTaskExecutor — general purpose (2–5 threads, queue of 1000)
  • sequentialTaskExecutor — single thread for ordered execution
  • lowPriorityTaskExecutor — background work (2 threads)
  • externalPriorityTaskExecutor — external service calls (2–5 threads)

Key Scheduled Jobs

Notifications email, statistics updates, search indexing (Lucene), REST token cleanup, reminders, video transcoding, assessment evaluation, lifecycle management, and more — all configured as Quartz triggers in schedulerContext.xml.

20. Upgrade Infrastructure (org.olat.upgrade)

OpenOlat includes a built-in migration framework in the org.olat.upgrade package that handles both database schema changes and data transformations when upgrading between versions. The system is designed to run automatically on startup, track progress persistently, and support clustered deployments.

Two-Phase Upgrade Process

The upgrade system operates in two distinct phases:

  1. Database schema upgrades — Executed early during Spring context initialization by the DatabaseUpgradeManager. These run SQL ALTER scripts (located in /database/mysql/alter_*.sql and /database/postgresql/alter_*.sql) to bring the database schema up to date. This happens before most services and modules are initialized, ensuring the schema is ready for application code.
  2. Post-system-init upgrades — Executed after the entire application (all Spring beans, modules, and services) has fully initialized. The UpgradeManager listens for the FrameworkStartedEvent and then runs data migration logic in a background thread via TaskExecutorManager. These upgrades can use any service or manager through @Autowired injection.
Important: Post-system-init upgrades run after all services and modules have been initialized. This means that any upgrade that changes module configurations (e.g., enabling features, modifying default settings in an AbstractSpringModule subclass) must be aware that the module has already been initialized with its previous configuration. If the module reads configuration during init(), changes made by the upgrader will only take effect on the next startup. In such cases, the upgrader may need to trigger a re-initialization of the affected module, or the module itself must be designed to pick up configuration changes dynamically.

Writing an Upgrade

Each upgrade class follows a standard pattern with idempotent sub-tasks:

public class OLATUpgrade_20_3_0 extends OLATUpgrade {
    private static final String VERSION = "OLAT_20.3.0";
    private static final String MIGRATE_SETTINGS = "MIGRATE SETTINGS";

    @Autowired
    private MyService myService;

    @Override
    public String getVersion() { return VERSION; }

    @Override
    public boolean doPostSystemInitUpgrade(UpgradeManager upgradeManager) {
        UpgradeHistoryData uhd = upgradeManager.getUpgradesHistory(VERSION);
        if (uhd == null) {
            uhd = new UpgradeHistoryData();
        } else if (uhd.isInstallationComplete()) {
            return false;  // Already done
        }

        boolean allOk = true;
        allOk &= migrateSettings(upgradeManager, uhd);

        uhd.setInstallationComplete(allOk);
        upgradeManager.setUpgradesHistory(uhd, VERSION);
        return allOk;
    }

    private boolean migrateSettings(UpgradeManager upgradeManager, UpgradeHistoryData uhd) {
        if (uhd.getBooleanDataValue(MIGRATE_SETTINGS)) {
            return true;  // Sub-task already done
        }
        // ... perform migration using @Autowired services ...
        uhd.setBooleanDataValue(MIGRATE_SETTINGS, true);
        upgradeManager.setUpgradesHistory(uhd, VERSION);
        return true;
    }
}

Key patterns:

  • Idempotent sub-tasks — Each migration step checks a boolean flag in UpgradeHistoryData before executing, allowing safe resume after interruption.
  • Service injection — Upgrades are Spring beans and can @Autowired any service.
  • Batch processing — Large data migrations should process in batches with intermediateCommit().
  • Version constant — Format is OLAT_XX.Y.Z (e.g., OLAT_20.3.0).

State Tracking

Upgrade progress is persisted as XStream-serialized XML files in the olatdata/system/ directory:

  • installed_upgrades.xml — Tracks post-system-init upgrades
  • installed_database_upgrades.xml — Tracks database schema upgrades

Each upgrade version maps to an UpgradeHistoryData object containing an installationComplete flag and a map of sub-task completion flags.

Cluster Awareness

In a clustered deployment, only the singleton node (configured via cluster.singleton.services=enabled) runs upgrades. Other nodes use UpgradeManagerDummy, which is a no-op implementation. This prevents concurrent upgrade execution across cluster nodes.

21. GUI Preferences

OpenOlat provides a per-user preference storage system for persisting UI state across sessions — such as table column visibility, selected segments, and view modes. This uses the Preferences interface and the GuiPreferenceService.

Preferences Interface

// Read a preference:
Preferences prefs = ureq.getUserSession().getGuiPreferences();
String value = (String) prefs.get(MyController.class, "selectedTab");

// Write a preference (persists immediately):
prefs.putAndSave(MyController.class, "selectedTab", "overview");

The key is a combination of the class (acting as namespace) and a string key. Values are serialized via XStream and stored in the o_gui_prefs database table. Any serializable object can be stored, though simple types (String, Integer, Boolean) are preferred.

GuiPreferenceService

GuiPreferenceService is the database-backed storage service. It provides direct CRUD operations and is used internally by the Preferences implementation. For most controllers, the Preferences interface (accessed via ureq.getUserSession().getGuiPreferences()) is sufficient.

Common Uses

  • FlexiTable column preferences — users can show/hide table columns; the selection is persisted automatically by the FlexiTable component using preferences
  • SegmentView selection — remembers which tab/segment the user last selected
  • View mode — table vs. card view, compact vs. expanded layouts
  • Filter state — last-used filter settings in list views

22. Internationalization (i18n)

Core Architecture

Core classes:

  • I18nModule (org.olat.core.util.i18n) — Spring module, manages language discovery, configuration, overlay/locale setup, gender strategies
  • I18nManager — Singleton service for key resolution, property loading, caching, recursive reference resolution
  • PackageTranslator (org.olat.core.gui.translator) — Per-controller translator bound to a specific package and locale, with fallback chain to org.olat.core and org.olat bundles

Translation files follow the colocated resource pattern: each UI package contains an _i18n/ directory with LocalStrings_XX.properties files (Java .properties format: key=value, # comments, \: escapes colons).

Statistics: ~382 i18n bundles, ~29,000 English keys, 36 language variants.

Locale Resolution Order

The I18nManager.getLocalizedString() method resolves translations through this fallback chain (overlay is checked first at every step):

Overlay Mechanism (Client Customization)

The overlay system allows clients to customize any translation key without modifying source code. Overlay files go in:

{userData}/customizing/lang/overlay/{package}/_i18n/LocalStrings_XX__customizing.properties

The overlay is checked first at every step of the fallback chain. A client can override any translation — product name, button label, help text — by placing a single properties file in the overlay directory. The overlay name customizing is configured in I18nModule.

Recursive Key Resolution (Cross-Referencing)

Translation values can reference other translation keys using two syntaxes:

  1. Same-package reference: $\:other.key — resolves other.key from the same .properties file
  2. Cross-package reference: $org.olat.other.package:other.key or ${org.olat.other.package:other.key} — resolves from a different package

Resolution is recursive up to 10 levels (guarded by recursionLevel counter). The regex pattern used: \$\{?([package]):([key])\}? (see resolvingKeyPattern in I18nManager).

When caching is enabled, references are resolved at property load time and cached. When caching is disabled (dev mode), resolution happens at access time. A referencingBundlesIndex tracks which bundles reference which keys for cache invalidation.

Currently ~851 same-package and ~1,171 cross-package references are in use across the codebase.

Fallback Bundle Chain

PackageTranslator first looks in its primary bundle, then falls back to:

  • org.olat.core (core fallback bundle) — common framework translations
  • org.olat (application fallback bundle) — common application translations

Uses java.text.MessageFormat: {0}, {1} etc. Single quotes have special meaning in MessageFormat and must be escaped as ''.

OpenOlat uses SASS-based theming with a layered architecture. Themes live in src/main/webapp/static/themes/ and are served via the /raw/ dispatcher with versioned URLs for cache-busting.

Theme Directory Structure

│   ├── theme.css           ← compiled output (committed)

Theme Layering

The light theme is the base — it includes Bootstrap integration and defines all SCSS variables. Product themes (like openolat) extend it by overriding variables in their _config.scss. Themes are compiled with the compiletheme.sh script, which wraps the sass CLI:

cd src/main/webapp/static/themes

Clients can create fully custom themes stored outside the codebase in a configurable guiCustomThemePath directory. The Theme class automatically renders all HTML header elements: CSS link, optional custom JS, favicons (multiple sizes), Apple touch icons, and PWA manifest. In templates: $r.staticThemeLink("image.png") resolves to the current theme's path.

24. Caching Infrastructure

OpenOlat uses Infinispan as its caching layer, accessed through the Coordinator.getCacher() facade. Caches store frequently accessed data (user profiles, course structures, compiled templates) in memory to avoid repeated database or filesystem access.

Obtaining a Cache

// In a Spring-managed service (typical pattern):
@Autowired
private CoordinatorManager coordinatorManager;

private CacheWrapper<Long, CourseData> courseCache;

@PostConstruct
public void init() {
    courseCache = coordinatorManager.getCoordinator().getCacher()
        .getCache("CourseService", "courses");
}

CacheWrapper API

The CacheWrapper<U, V> interface provides a simple map-like API:

cache.get(key)                          // retrieve
cache.put(key, value)                   // store (non-invasive reload from DB)
cache.update(key, value)                // store (propagates invalidation in cluster)
cache.putIfAbsent(key, value)           // atomic conditional store
cache.computeIfAbsent(key, k -> load()) // lazy loading pattern
cache.remove(key)                       // explicit invalidation
cache.clear()                           // flush entire cache

Important distinction: put() is for reloading data from a persistent source (DB/filesystem). update() is for data that changed and should trigger invalidation events in a cluster.

Cache Configuration

Caches are configured in infinispan-config.xml. Pre-defined caches have specific sizes and TTLs; dynamically created caches get defaults (10,000 entries, 15-minute idle timeout). Key caches:

CacheMax EntriesMax IdleContent
CourseFactory-courses1,5001 hourFull course structures
UserManager-username20,00045 minUsername lookups
Velocity-templates7,700NeverCompiled Velocity templates
QTIWorks-testSessionControllers50,0002 hoursQTI test execution state
VFSLockManager-file-locks50,0006 hoursWebDAV file locks

Cache Invalidation

Caches are invalidated through three mechanisms:

  1. TTL-based expiration — Infinispan's reaper thread removes idle entries (configurable per cache)
  2. Event-driven invalidation — When data changes, fire a MultiUserEvent via the EventBus. Listeners on all cluster nodes call cache.remove(key) to invalidate stale entries.
  3. Manual flush — Administrators can clear individual caches via the admin UI (AllCachesController)
// Event-driven invalidation pattern:
// 1. After modifying a course, fire event:
MultiUserEvent event = new ModifyCourseEvent(courseId);
CoordinatorManager.getInstance().getCoordinator().getEventBus()
    .fireEventToListenersOf(event, courseOres);

// 2. Listener on each node removes from cache:
public void event(Event event) {
    if (event instanceof ModifyCourseEvent mce) {
        courseCache.remove(mce.getCourseId());
    }
}

25. Logging Infrastructure

OpenOlat has two distinct logging layers: technical logging (standard log files) and user activity logging (database-persisted audit trail).

Technical Logging (Log4j2)

All technical logging uses Log4j2 via the Tracing factory class:

private static final Logger log = Tracing.createLoggerFor(MyClass.class);

log.info("Course published: {}", courseId);
log.warn("Retry failed for resource: {}", resourceKey);
log.error("Unexpected error in service", exception);

Tracing.createLoggerFor() returns a standard SLF4J Logger. The Tracing class also sets up MDC (Mapped Diagnostic Context) entries for request correlation:

  • ThreadLocalUserActivityLogger adds user identity, IP address, and session ID to MDC
  • Log4j2 configuration is in src/main/resources/log4j2.xml
  • A special audit marker Tracing.M_AUDIT is available for log statements that should be written to the audit log appender

User Activity Logging

For compliance and auditing, OpenOlat records user actions in the o_loggingtable database table via ThreadLocalUserActivityLogger. This captures structured events: who did what, on which resource, at what time.

// Log a user action (from within a controller):
ThreadLocalUserActivityLogger.log(LearningResourceLoggingAction.LEARNING_RESOURCE_OPEN, getClass(),
    LoggingResourceable.wrap(repositoryEntry));

Logging actions are defined as enum-like classes (e.g., LearningResourceLoggingAction, CourseLoggingAction) that describe the action type. LoggingResourceable wraps the business objects that were affected. The logging system is decoupled from the business logic — it runs as a background write and never blocks the user interaction.

Best Practices for Logging

  • Always use Tracing.createLoggerFor(), never instantiate loggers directly
  • Use parameterized messages (log.info("msg: {}", val)) to avoid string concatenation when logging is disabled
  • Use log.error(msg, exception) for exceptions — always pass the exception as the last argument to preserve the stack trace
  • Do not log sensitive data (passwords, tokens, personal data)
  • Use ThreadLocalUserActivityLogger for business-relevant actions that need to be auditable
  • Use log.isDebugEnabled() guards only when constructing the log message is expensive

27. Identity, Security & Organisations

OpenOlat's security model is built around three core concepts: Identity (who you are), Roles (what you can do), and Organisation (where you belong). These are defined in the org.olat.basesecurity and org.olat.core.id packages.

Identity

Identity is the central authentication entity. It represents a unique account in the system and is the primary key used for access control, group membership, and audit trails. Key properties:

  • getKey() — unique database ID (Long)
  • getName() — login username (unique, immutable after creation)
  • getUser() — link to the User profile entity (one-to-one, see section 28)
  • getStatus() — account lifecycle state: activ, permanent, pending, inactive, login_denied, deleted
  • getExternalId() — external system identifier (LDAP DN, Shibboleth ID, etc.)
  • getLastLogin() — timestamp of last successful authentication

Identity is always accessed through BaseSecurity (the service interface) or obtained from the UserRequest via ureq.getIdentity().

Roles

Roles are defined as enums and fall into two categories:

EnumScopeValues
OrganisationRolesSystem-wide / per organisationsysadmin, administrator, usermanager, rolesmanager, groupmanager, learnresourcemanager, poolmanager, curriculummanager, lecturemanager, qualitymanager, linemanager, principal, author, user, invitee, guest
GroupRolesPer resource / group / courseowner, coach, participant, invitee, waiting
CurriculumRolesPer curriculum / curriculum elementcurriculummanager, curriculumowner, curriculumelementowner, owner, mastercoach, coach, participant
ProjectRolePer projectowner, leader, projectOffice, participant, supplier, client, steeringCommitee, invitee
ContentRolesPer content editor element (portfolio, form)owner, coach, reviewer, invitee, readInvitee
Course roles: Courses use GroupRoles for their membership model. A course has an internal Group where owner = course owner, coach = course coach/tutor, and participant = course participant. The same GroupRoles enum is also used for business groups, learning resource memberships, and curriculum element memberships.

The Roles object is an immutable container attached to each user session. It holds a list of RolesByOrganisation entries (one per organisation the user belongs to) and provides query methods:

Roles roles = ureq.getUserSession().getRoles();
boolean isAdmin = roles.isAdministrator();           // any org
boolean isAuthor = roles.isAuthor();                  // any org
boolean hasRole = roles.hasRole(org, OrganisationRoles.usermanager);
List<OrganisationRef> orgs = roles.getOrganisationsWithRole(OrganisationRoles.author);

Organisations

Organisations form a hierarchical tree that models the institutional structure (university → faculty → department). Each organisation has:

  • getKey() / getIdentifier() — unique ID and human-readable identifier
  • getParent() — parent organisation (null for the root)
  • getMaterializedPathKeys() — slash-delimited path of ancestor keys (e.g., /1/42/103/) enabling efficient subtree queries via SQL LIKE
  • An internal Group entity that holds all membership relations

Role assignments are always scoped to an organisation. A user can be an administrator in one organisation and just an author in another. The OrganisationService manages memberships:

organisationService.addMember(organisation, identity, OrganisationRoles.author);
organisationService.removeMember(organisation, identity, OrganisationRoles.author);
List<Identity> authors = organisationService.getMembersIdentity(organisation, OrganisationRoles.author);

Group Membership & Inheritance

At the database level, all memberships are stored as GroupMembership records linking an Identity, a Group, and a role string. The inheritanceModeString field controls how roles propagate in the organisation tree:

  • none — role is valid only for this organisation
  • root — role was directly assigned here and inherits to children
  • inherited — role was propagated from a parent organisation

Grant Mechanism

The Grant entity provides fine-grained permissions on specific OLATResource instances. A grant links a group role to a permission string on a resource. This is used for resource-level access control (e.g., "coaches of this course group can edit the course").

BaseSecurity Service

BaseSecurity is the primary service interface for identity and authentication management:

  • findIdentityByLogin(login) — look up by username
  • loadIdentityByKey(key) — load by database ID
  • createAndPersistIdentityAndUser(...) — create a new account
  • getRoles(identity) — get the immutable Roles object
  • isIdentityPermittedOnResourceable(identity, permission, ores) — check resource-level permission
  • getAuthentications(identity) — list authentication providers (OLAT, LDAP, OAuth, etc.)

28. User Properties & Profile

While Identity handles authentication and access control, the User entity (from org.olat.core.id and org.olat.user) stores profile data: name, email, institutional affiliation, and other personal attributes.

Identity vs. User

Identity and User have a one-to-one relationship. Identity owns the reference: identity.getUser() returns the User. The User entity (UserImpl) is a JPA entity with 53 mapped database columns for property storage, plus a generic userProperties map for custom properties.

UserPropertyHandler

User properties are not accessed by direct getter/setter methods. Instead, OpenOlat uses a dynamic property handler system where each property (first name, email, phone, etc.) is represented by a UserPropertyHandler implementation. The handler interface provides:

  • getName() — property key (e.g., "firstName", "email", "institutionalName")
  • getUserProperty(user, locale) — get the formatted value
  • setUserProperty(user, value) — set the raw value
  • addFormItem(locale, user, key, isAdmin, container) — create a form element for editing this property
  • updateUserFromFormItem(user, formItem) — write form value back to the user
  • isValid(user, formItem, properties) — validate the property value
  • getColumnDescriptor(colIndex, action, locale) — create a table column descriptor for this property

This design allows each property to control its own rendering, validation, and display format. For example, EmailProperty validates email format, DatePropertyHandler renders a date picker, and GenderPropertyHandler renders a dropdown.

Context-Based Visibility

UserPropertyUsageContext controls which properties appear in which UI contexts. Each context (identified by a string name like "org.olat.user.ProfileFormController" or "org.olat.admin.user.UsermanagerUserSearchForm") defines:

  • Which properties are visible (ordered list)
  • Which are mandatory
  • Which are admin-only (visible only to user managers)
  • Which are read-only for the user

The UserPropertiesConfig (configured via Spring XML in olatUserProperties.xml) registers all available handlers and defines contexts. Controllers query this configuration to build forms:

UserPropertiesConfig propConfig = CoreSpringFactory.getImpl(UserPropertiesConfig.class);
List<UserPropertyHandler> handlers = propConfig.getUserPropertyHandlersFor(
    "org.olat.user.ProfileFormController", isAdmin);
for (UserPropertyHandler handler : handlers) {
    handler.addFormItem(getLocale(), identity.getUser(), usageIdentifier, isAdmin, formLayout);
}

User Properties in Tables

The same handler system integrates with FlexiTable. Each handler can create a FlexiColumnModel for its property, which means table columns for user properties are dynamically constructed based on context configuration. This ensures that the same property renders consistently across all views (search results, member lists, admin tables).

UserManager

UserManager is the primary service for user profile operations. It provides display name resolution with an internal cache to avoid repeated database lookups:

  • getUserDisplayName(identity) — formatted display name (configurable format: "First Last" or "Last, First")
  • getUserDisplayNamesByKey(identityKeys) — batch lookup returning a Map<Long, String>
  • updateUserFromFormItems(identity, handlers, formItems) — bulk update from a form

29. XSS & Security Infrastructure

OpenOlat employs multiple layers of defense against cross-site scripting (XSS) and other injection attacks.

OWASP AntiSamy (HTML Sanitization)

User-generated HTML content (from rich-text editors, imported HTML pages, etc.) is sanitized through OWASPAntiSamyXSSFilter, which uses a whitelist-based policy defined in OpenOLATPolicy.java. The policy allows safe HTML tags and attributes while stripping everything else — including <script>, event handlers (onclick, onerror), and dangerous URI schemes (javascript:).

// Sanitize user HTML input:
OWASPAntiSamyXSSFilter filter = new OWASPAntiSamyXSSFilter();
String safeHtml = filter.filter(userInput);

// The filter is applied automatically to:
// - RichTextElement content on form submission
// - Imported HTML pages in course elements
// - HTML content stored via REST API

Output Escaping

For non-HTML contexts, the framework provides manual escaping utilities:

  • $r.escapeHtml($text) / StringHelper.escapeHtml() — HTML entity encoding for text content
  • $r.escapeJavaScript($text) — escapes strings for safe embedding in JS
  • StringHelper.xssScan() — quick scan for dangerous patterns (used for validation, not sanitization)

Velocity templates that render user data must escape output. The $r.render() method is safe (components handle their own escaping), but contextPut() variables rendered directly via $myVar are not auto-escaped — use $r.escapeHtml($myVar) for user-provided text.

CSRF Protection

The Form class generates a unique CSRF token per form instance, embedded as a hidden field. On submission, Form.validate() checks the token before processing. The token is tied to the user's window, preventing cross-origin form submissions.

Security Headers

HeadersFilter (a servlet filter) adds security headers to every response:

  • X-Frame-Options: SAMEORIGIN — prevents clickjacking
  • X-Content-Type-Options: nosniff — prevents MIME-type sniffing
  • X-XSS-Protection: 1; mode=block — legacy XSS filter
  • Content-Security-Policy — configured via CSPModule, controls allowed script/style/media sources
  • Strict-Transport-Security — enforces HTTPS

Choosing the Right Controller Type

Creating a New View (Checklist)

  1. Create the controller — extend BasicController or FormBasicController
  2. Create the Velocity template — in _content/ next to the controller package
  3. Add i18n keys — in _i18n/LocalStrings_en.properties (and other locales)
  4. Wire it up — create the child in the parent, call listenTo(), embed via $r.render()

Key Lifecycle Rules

  • Always listenTo() child controllers — ensures automatic disposal and event routing
  • Always removeAsListenerAndDispose() before replacing a controller reference
  • Never store UserRequest — it is valid only for the current request cycle
  • Use DBFactory.getInstance().commitAndCloseSession() in background tasks
  • Use CoreSpringFactory.getImpl() for service access in non-Spring-managed classes
  • Use Tracing.createLoggerFor() for all logging — never instantiate loggers directly. Use parameterized messages. See section 25
  • Use ThreadLocalUserActivityLogger.log() for business-relevant actions that need an audit trail

WindowControl — Modals & Overlays

// Push a modal dialog
getWindowControl().pushAsModalDialog(editCtrl.getInitialComponent());

// Remove when done
getWindowControl().removeModalDialog(editCtrl.getInitialComponent());

// Show a callout popup near an element
getWindowControl().pushAsCallout(helpCtrl.getInitialComponent(), targetDomId, settings);

// Full-screen takeover
getWindowControl().pushFullScreen(editorCtrl, "o_editor_body");

Module Package Structure

Well-structured modules (e.g., certificationprogram, grading, topicbroker, curriculum) follow a consistent package layout:

org.olat.modules.mymodule/
├── MyModule.java              // AbstractSpringModule — config properties
├── MyService.java             // public service interface
├── manager/
│   └── MyServiceImpl.java     // @Service implementation with DAO injection
│   └── MyDAO.java             // DB access, package-private
├── model/
│   └── MyEntity.java          // JPA @Entity
│   └── MyEntityImpl.java      // implementation (if interface exists)
├── ui/
│   ├── MyListController.java  // FlexiTable-based list
│   ├── MyEditController.java  // FormBasicController for editing
│   ├── _content/              // Velocity templates
│   └── _i18n/                 // LocalStrings_XX.properties
├── restapi/
│   └── MyWebService.java      // JAX-RS endpoints
└── _spring/
    └── mymoduleContext.xml     // optional Spring XML config

Key conventions from best-practice modules:

  • Service interface + impl separation — public interface in the package root, @Service implementation in manager/. DAOs are package-private, never used directly by UI code.
  • Module class for feature toggles — AbstractSpringModule subclass manages isEnabled() and persisted configuration properties.
  • FlexiTable for all lists — data source pattern with server-side sorting/filtering. Use DefaultFlexiTableDataModel for small datasets, FlexiTableDataSourceDelegate for large ones.
  • Breadcrumb navigation — use BreadcrumbedStackedPanel for multi-level drill-down views. Push controllers onto the stack, handle Event.BACK_EVENT.
  • Toolbar actions — add buttons via toolbarPanel.addTool() for create/export/import actions at the top of list views.
  • Security callbacks — pass permission objects (e.g., GradingSecurityCallback) to controllers rather than checking roles inline.
