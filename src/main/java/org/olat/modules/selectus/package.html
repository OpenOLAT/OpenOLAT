Controllers hierarchy for the staff and committee

A staff member is someone with author permissions.

The first panel contains the list of positions with a list of filters in the the toolbar. This list of filters is controlled by the permissions the user. A committee member can see only the applications in screening ("screening" and "closed and in screening"), a staff member can is all of them.

The controller is: {@link org.olat.modules.selectus.ui.PositionListController}

From the list of positions, can committee and staff members select a position to see all the details. A Controller ({@link org.olat.modules.selectus.ui.PositionController}) manages three sub-controllers with a segment view. These controllers are:

  • Informations about the position (segment "Details"): {@link org.olat.modules.selectus.ui.PositionDetailsController}
  • List of applications (segment "Applications"): {@link org.olat.modules.selectus.ui.PositionApplicationsController}. The datamodel for the list is the same as for the Excel export ({@link org.olat.modules.selectus.ui.PositionApplicationsExcelDataModel}).
  • List of committee member (segment "Committee"): {@link org.olat.modules.selectus.ui.PositionCommitteeController}

From the list of applications, can we see the details of a selected application. A controoler manages with a segment view two sub-controllers:

  • The details about the applicant (segment "Details"): {@link org.olat.modules.selectus.ui.ApplicationDetailsController}
  • The list of documents uploaded by the applicant (segment "Documents"): {@link org.olat.modules.selectus.ui.ApplicationDocumentsController}

Add a position

The link in the toolbar "Add position" in {@link org.olat.modules.selectus.ui.PositionListController} open a popup window where a staff member can fill the informations about the position. The controller is {@link org.olat.modules.selectus.ui.PositionEditController}

Add an application

The wizard to add a new application is useb by staff members and by the applicants. The code is the same with severals switchs to diffenciate them at some points. The steps are in the package: {@link org.olat.modules.selectus.ui.app_wizard}. Every step have their own class, a step controller and the effective controller to grab the informations. This way can the controllers be reused across the application. The steps are (I give only the controllers which do the job):

  • Choose position (only for staff members): {@link org.olat.modules.selectus.ui.app_wizard.ChoosePositionController}
  • Information about the applicant: {@link org.olat.modules.selectus.ui.app_wizard.EditPersonController}
  • Academical background: {@link org.olat.modules.selectus.ui.app_wizard.AcademicalBackgroundController}
  • Documents upload: {@link org.olat.modules.selectus.ui.app_wizard.DocumentsController}. The size of the documents are limited to 5MB.
  • Review and submit: {@link org.olat.modules.selectus.ui.app_wizard.ReviewAndSubmitController}

Add a committee

The wizard has one or two steps and is located in the package: {@link org.olat.modules.selectus.ui.committee_wizard}

  • Search with e-Mail in the LDAP directories. If the user exists, import it in the recruiting tool or if already imported, update it. In this case, the wizard is closed. If the user cannot be found, go to step 2. The controller is {@link org.olat.modules.selectus.ui.committee_wizard.EmailStepController}.
  • The second allows to create a new user in the recruiting tool without any permissions but a login. The controller is {@link org.olat.modules.selectus.ui.committee_wizard.MembersStepController}. A password is saved as a property in plain text and as authentication. An e-mail is send to the staff with the username and password.

Special cases

The possible documents

The list of documents is defined in an enum: {@link org.olat.modules.selectus.ui.DocumentEnum}. Methods in the enum allows to retrive and save the binary document from the application.

Excel list of applications

The output is done with JExcelApi, a small library to generate Excel files (not CSV). The datamodel is shared with the application list controller: {@link org.olat.modules.selectus.ui.PositionApplicationsExcelDataModel}.

The generator is embbeded in {@link org.olat.modules.selectus.ui.JExcelTableExport}.

PDF list of applications

The PDF is generated by FOP. A cusotm data model is used: {@link org.olat.modules.selectus.ui.PositionApplicationsPDFDataModel} and transformed in a DOM. This DOM is transformed with XSLT and given to FOP. The exported is {@link org.olat.modules.selectus.ui.FOPTableExport}.

Combined documents (PDF)

The documents uploaded by an applicant can be comined in a single PDF. The utility class which does the job is {@link org.olat.modules.selectus.ui.PDFMergeUtility}. It's a copy from the utility with the same name from PDFBox with some changes. The changes allows to add intermediate pages generates with FOP.

The table of content of all the documents are merged in one big TOC.

Login

The login is done with a special controller which solves two different probles:

  • Custom design without guest link (but guest account are on for applicatant)
  • Authenticate in a single request against LDAP first and Olat.

The controller is: {@link org.olat.modules.selectus.ui.login.RecruitingAuthenticationController}. It's loaded by a custom ChiefCreator and defined in the Spring configuration.

The rating component

The rating component in package {@link org.olat.modules.selectus.ui.rating} is an adaptation of the user rating component for blog and podcast with custom renderer and comparator.

Segment view, toolbar and bread crumps

The package {@link org.olat.modules.selectus.ui.main} contains some classes which can be merged in the main code base of Olat. It contains the segment view, a toolbar and a slightly enhanced version of the BreadCrumps.

The DB model

It use the standard model from olatcommons and three new tables for position, application and the document attached to an application. The model is very flat.

The model is served by two managers: {@link org.olat.modules.selectus.ui.manager.PositionManagerImpl} and {@link org.olat.modules.selectus.ui.manager.ApplicationManagerImpl} which serve mostly as DAO. And the controllers use a frontend manager (exclusively) {@link org.olat.modules.selectus.ui.manager.RecruitingFrontendManagerImpl} which implemented more business logic.

There are three tables: fx_r_position, fx_r_application, fx_r_attachment.

  • fx_r_position: the position
  • fx_r_application: all the informations of an application excepted the uploads files
  • fx_r_attachment: an uploaded file

The Hibernate mapping

The mapping is in package {@link org.olat.modules.selectus.model}

  • fx_r_position: the table is mapped to PositionImpl.
  • fx_r_application: the table is mapped to a main class {@link org.olat.modules.selectus.model.ApplicationImpl} which embbed several classes has components for a more object oriented way. these embbeded classes are {@link org.olat.modules.selectus.model.AddressImpl}, {@link org.olat.modules.selectus.model.PersonImpl}, {@link org.olat.modules.selectus.model.AcademicalBackgroundImpl} and a {@link org.olat.modules.selectus.model.AttachmentImpl} per file.
  • fx_r_attachment: the table is mapped to two classes: {@link org.olat.modules.selectus.model.AttachmentImpl} to hold only the file information without the file and the {@link org.olat.modules.selectus.model.AttachmentDataImpl} which hold the file itself.

Cron job

A cron job ({@link org.olat.modules.selectus.ui.manager.CloseStatusJob}) check if a position must be closed every minute. It's configured in spring.

Configuration

There are two spring files for the configuration. The {@link org.olat.modules.selectus.recruitingContext.xml} contains all the configuration of the recruiting tool and the {@link org.olat.modules.selectus.application.xml} all the overriden configuration of olatcore and olatcommons.

Log files

Weblogic doesn't support the standard way of Olat the start the logger of Spring. I added a custom one in a servlet listener which load a configuration file recruiting_log4j.xml which doesn't conflict with Olat and Weblogic.
