Overview

The org.olat.core.util.docxToMarkdown package implements a zero-dependency DOCX-to-Markdown converter for OpenOlat. It reads DOCX files (ZIP archives conforming to the OOXML standard), extracts text, formatting, images, shapes, math, and metadata, and produces clean Markdown output with media files on disk.

Design Principles

  • Stateless @Service — all conversion state is local to each convert() call. Thread-safe by design.
  • SAX streaming — the main document.xml is streamed directly from the ZIP to a SAX parser. No DOM tree is built.
  • Security-first — XXE, zip bombs, zip slip, macros, and JavaScript URLs are all blocked at the entry point.
  • Graceful degradation — unsupported features produce warnings, never exceptions. The converter always returns a result.
Key Principle: Memory usage is proportional to text content, not file size. A 1 GB DOCX with many images uses only ~35 MB of heap because images and videos are streamed to disk.

Public API

Only 3 public classes. Everything else is package-private.

DocxToMarkdownService

Spring @Service entry point. Registered via component-scan in utilCorecontext.xml.

@Service
public class DocxToMarkdownService {
    /** Convert from a java.io.File */
    public DocxToMarkdownResult convert(File docxFile);
    /** Convert from a VFSLeaf (local files accessed directly, non-local copied to temp) */
    public DocxToMarkdownResult convert(VFSLeaf vfsLeaf);
}

DocxToMarkdownResult

Conversion result record.

public record DocxToMarkdownResult(
    String markdown,          // front matter + body, never null
    File basePath,            // temp dir containing media/ subdirectory (may be null)
    List<DocxConversionMessage> messages
) {
    public boolean hasMessages();
    /** HTML <ul><li>...</li></ul> — HTML-escaped, safe for UI display */
    public String renderMessagesAsHtml(Translator translator);
    /** Plain text "- message\n- message" — for logging or non-HTML contexts */
    public String renderMessagesAsText(Translator translator);
}

DocxConversionMessage

A single conversion message with i18n key and substitution arguments. The service never produces user-visible strings — only i18n keys.

public record DocxConversionMessage(
    Level level,    // INFO, WARNING, ERROR
    String i18nKey, // e.g. "docx.convert.warn.image.skipped"
    String[] args   // substitution arguments for {0}, {1}, ...
) {
    public String translate(Translator translator);
}

Usage Example

@Autowired
private DocxToMarkdownService docxService;

// From a VFS leaf (preferred in OpenOlat)
DocxToMarkdownResult result = docxService.convert(vfsLeaf);
// Or from a java.io.File
DocxToMarkdownResult result = docxService.convert(docxFile);
String markdown = result.markdown();
File mediaDir = result.basePath(); // contains media/ subdirectory with images

if (result.hasMessages()) {
    String warnings = result.renderMessagesAsText(translator);
    log.info("Conversion messages:\n{}", warnings);
}

// Clean up temp directory when done
FileUtils.deleteDirsAndFiles(result.basePath(), true, true);

Pipeline Architecture

The converter operates as a 6-stage pipeline. Stages 1–5b parse auxiliary XML parts into lookup maps. Stage 6 streams document.xml through a SAX handler that references these maps.

6-stage SAX pipeline. document.xml is streamed directly from the ZIP to the handler without buffering.
Streaming design: Stages 2–5b parse small auxiliary XML files into byte arrays (total <1 MB). Stage 6 streams document.xml directly from the open ZipFile to the SAX parser, keeping memory constant regardless of document size.

Image Dimension Mapping

Image dimensions come from wp:extent in the OOXML (EMU units), representing the display size in the Word document layout — not the native image pixel resolution. A standard Word page content area is ~605px wide at 96 DPI (~16cm). The importer maps image width as a proportion of the Word page width to the content editor's ImageSize CSS class:

Word page proportionPixel range (at 605px page)ImageSizeCSS width
≤ 35%≤ 212pxsmall25%
35–55%213–333pxmedium40%
55–80%334–484pxlarge60%
> 80%> 484pxfill100%

Integration

From a Controller

  1. Add @Autowired DocxToMarkdownService docxService;
  2. Call docxService.convert(file) with the uploaded .docx file
  3. Use result.markdown() for the converted text
  4. Use result.basePath() to access extracted media files (in media/ subdirectory)
  5. Clean up result.basePath() when done — it is a temp directory
@Autowired
private DocxToMarkdownService docxService;

public void handleUpload(File uploadedFile) {
    DocxToMarkdownResult result = docxService.convert(uploadedFile);

    if (result.hasMessages()) {
        for (DocxConversionMessage msg : result.messages()) {
            if (msg.level() == Level.ERROR) {
                showError(msg.translate(getTranslator()));
                return;
            }
        }
    }

    String markdown = result.markdown();
    File mediaDir = new File(result.basePath(), "media");
    // ... process markdown and media files ...

    // Clean up when done
    FileUtils.deleteDirsAndFiles(result.basePath(), true, true);
}

Spring Context

The service is registered via component-scan in utilCorecontext.xml. No additional XML configuration, no @Bean declarations, and no init/destroy lifecycle methods are needed.
